# 产品需求文档（PRD）：「知链」—— AI驱动的个人知识图谱


## 一、产品概述

### 1.1 产品定位

**「知链」（暂定名）** 是一款AI驱动的个人知识图谱管理工具，帮助用户将碎片化的笔记、文档、想法自动转化为可视化的知识网络，让知识不再是"沉睡的文档"，而是可探索、可连接的"活地图"。

### 1.2 核心价值主张

> **输入任何内容，AI帮你构建知识网络；拖拽任意节点，发现隐藏的知识关联。**

传统笔记工具（Notion、Obsidian等）需要用户手动建立双向链接，门槛高、维护成本大。而知链的核心差异在于：**AI自动完成从"非结构化内容"到"结构化图谱"的转化**，用户只需专注记录和探索。

### 1.3 差异化优势

| 维度 | 传统PKM工具 | 知链（MVP） |
|------|------------|------------|
| 图谱构建 | 手动建立[[双向链接]] | AI自动抽取实体与关系 |
| 内容录入 | 需结构化整理 | 任意文本/文档直接丢入 |
| 知识发现 | 依赖用户主动关联 | 图谱可视化 + AI推荐关联 |
| 使用门槛 | 高（需学习笔记法） | 低（零门槛上手） |


## 二、目标用户

### 2.1 核心用户画像

**知识工作者（25-40岁）**
- 产品经理、运营、技术研发等岗位
- 日常处理大量文档、会议纪要、项目复盘
- 痛点：信息碎片化，知识沉淀难复用

**终身学习者与学生**
- 备考、读研、自学新领域
- 痛点：知识点散落各处，无法形成体系

**内容创作者**
- 博主、作者、研究员
- 痛点：素材管理混乱，灵感难以串联

### 2.2 用户规模预估

参考Obsidian（超200万用户）、Logseq等工具的快速增长，以及Graphify上线48小时即获2k+ Star的热度，C端知识图谱工具市场需求明确且仍在增长中。


## 三、MVP核心功能

### 3.1 功能模块总览

```
┌─────────────────────────────────────────────────────┐
│                    知链 MVP                          │
├─────────────┬─────────────┬─────────────────────────┤
│  内容录入    │  图谱构建    │     探索与交互           │
├─────────────┼─────────────┼─────────────────────────┤
│ • 文本输入   │ • AI自动    │ • 力导向图谱可视化        │
│ • 文档导入   │   实体抽取   │ • 节点拖拽/缩放/筛选      │
│ • 网页链接   │ • AI关系    │ • 点击节点查看详情        │
│ • 浏览器插件 │   识别      │ • 全文搜索               │
│   (快速收藏) │ • 自动建图   │ • AI问答（图谱增强）      │
└─────────────┴─────────────┴─────────────────────────┘
```

### 3.2 功能详解

#### 模块一：内容录入（P0）

**F1.1 文本/笔记录入**
- 支持Markdown编辑与渲染
- 用户可直接粘贴或撰写笔记

**F1.2 文档导入**
- 支持PDF、TXT、Markdown文件上传
- MVP阶段支持单文件上传

**F1.3 网页链接收藏**（P1，可延后）
- 浏览器插件一键收藏网页
- 自动提取网页正文内容

#### 模块二：AI自动图谱构建（核心差异化，P0）

**F2.1 自动实体抽取**
- 用户录入内容后，后台自动调用LLM抽取核心实体（概念、人物、项目、技能等）
- 参考Graphify的全模态自动图谱化思路

**F2.2 自动关系识别**
- 识别实体间关系（如"属于"、"依赖"、"引用"、"相似于"等）
- 生成结构化的"实体-关系-实体"三元组

**F2.3 图谱自动构建**
- 将抽取的实体与关系自动写入图数据库
- 用户无需任何手动操作即可看到知识图谱

#### 模块三：图谱可视化与交互（P0）

**F3.1 力导向图谱视图**
- 使用力导向布局展示知识网络
- 节点用颜色区分不同类型（主题/人物/概念等）
- 支持画布缩放、拖拽平移

**F3.2 节点交互**
- 点击节点查看详情（关联内容、来源笔记）
- 悬停显示节点名称和关系标签

**F3.3 筛选与聚焦**
- 按主题/类型筛选节点
- 点击节点自动聚焦其邻域

**F3.4 全文搜索**
- 搜索关键词，高亮匹配节点

#### 模块四：AI问答（P1，MVP可简版）

**F4.1 图谱增强问答**
- 用户提问时，系统基于知识图谱检索相关内容
- 结合GraphRAG思路，让回答更精准、可追溯
- MVP阶段可先用简单检索 + LLM生成回答

### 3.3 非功能需求

- **本地优先**：数据默认存储在本地，保障隐私
- **响应速度**：图谱渲染<2秒，搜索<1秒
- **跨平台**：Web端优先（MVP），后续扩展桌面端

### 4.2 技术选型详解

#### 前端

| 技术 | 选型理由 |
|------|---------|
| **React + TypeScript** | 生态成熟，组件化开发效率高 |
| **Cytoscape.js** | 专为网络图可视化设计，API友好，性能优于D3.js手搭方案 |
| **Tailwind CSS** | 快速构建UI，保持设计一致性 |

#### 后端

| 技术 | 选型理由 |
|------|---------|
| **FastAPI (Python)** | 异步高性能，与AI生态无缝集成 |
| **WebSocket** | 支持图谱实时交互（拖拽、缩放等） |

#### 图数据库

| 技术 | 选型理由 |
|------|---------|
| **Neo4j** | 原生图数据库，Cypher查询成熟，社区生态完善 |

> **备选方案**：MVP阶段也可考虑更轻量的方案，如使用JSON文件 + 内存图结构，降低部署复杂度。

#### AI服务

| 组件 | 选型理由 |
|------|---------|
| **DeepSeek API** | 成本低，中文理解能力强 |
| **本地LLM (Ollama)** | 可选，满足隐私敏感用户 |

> **成本优化参考**：Graphify通过AST解析 + 并行LLM子代理实现了71.5倍的Token节省，MVP阶段可通过缓存机制和增量更新控制API成本。

### 4.3 核心数据模型

**节点 (Node)**
```
{
 id: string,
 name: string,
 type: 'concept' | 'person' | 'project' | 'note' | 'topic',
 sourceNoteId: string, // 来源笔记ID
 createdAt: timestamp,
 metadata: object
}
```

**关系 (Relationship)**
```
{
 id: string,
 source: string, // 源节点ID
 target: string, // 目标节点ID
 type: 'belongs_to' | 'depends_on' | 'references' | 'similar_to' | 'derives_from',
 sourceNoteId: string,
 confidence: number // AI置信度
}
```

**笔记 (Note)**
```
{
 id: string,
 title: string,
 content: string, // Markdown格式
 type: 'text' | 'doc' | 'link',
 createdAt: timestamp,
 updatedAt: timestamp,
 tags: string[]
}
```

### 4.4 AI图谱构建流程

```
用户录入内容
 ↓
文本预处理（分句、分词）
 ↓
LLM实体抽取 ← 提示词工程（定义实体类型）
 ↓
LLM关系识别 ← 提示词工程（定义关系类型）
 ↓
三元组生成与去重
 ↓
写入Neo4j + 触发前端图谱更新
```

**MVP提示词示例**：
> "请从以下文本中抽取核心实体（概念、人物、项目、技能）和它们之间的关系。输出格式为JSON：{'entities': [{'name':, 'type':}], 'relations': [{'source':, 'target':, 'type':}]}"


## 五、MVP开发计划

### 5.1 里程碑

| 阶段 | 周期 | 核心交付 |
|------|------|---------|
| **Phase 0: 技术预研** | 1周 | 技术选型确认、原型验证 |
| **Phase 1: 数据层** | 2周 | Neo4j schema设计、SQLite设计、文件存储 |
| **Phase 2: AI服务** | 2周 | 实体抽取API、关系识别API、提示词调优 |
| **Phase 3: 后端API** | 2周 | FastAPI接口、WebSocket、图谱查询 |
| **Phase 4: 前端MVP** | 3周 | 笔记录入、图谱渲染、基础交互 |
| **Phase 5: 联调测试** | 1周 | 端到端测试、性能优化 |
| **总计** | **11周** | 可上线Web MVP |

### 5.2 MVP功能优先级

| 优先级 | 功能 |
|--------|------|
| **P0（必须有）** | 文本录入、AI自动建图、图谱可视化、节点点击查看 |
| **P1（尽量有）** | 文档导入、全文搜索、筛选聚焦 |
| **P2（可延后）** | 浏览器插件、AI问答、多用户/同步 |

### 5.3 风险与应对

| 风险 | 应对策略 |
|------|---------|
| AI抽取准确率不足 | 提示词工程优化 + 用户可手动编辑节点/关系 |
| 大规模图谱性能问题 | 节点分页加载 + 邻域聚焦（默认不渲染全图） |
| API成本超预期 | 本地缓存 + 增量更新 + 提供本地LLM选项 |
| 用户上手困难 | 首次使用提供示例数据 + 引导式 onboarding |
